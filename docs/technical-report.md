# 技术报告

## 1. 项目目标

`courseweb-cli` 的目标，是把北大教学网里高频、重复、依赖浏览器的操作，整理成稳定可复用的命令行能力，同时保持足够真实的浏览器行为，保证登录和受保护资源访问能工作。

## 2. 当前范围

截至目前，项目已经覆盖：

- 浏览器登录
- 会话持久化
- 账号管理
- 课程发现与课程上下文
- 通知列表与详情
- 教学内容浏览与下载
- 作业列表与详情
- 安全默认的作业提交流程
- 课堂实录列表、详情与下载
- 适合 agent 的 `--json` 输出

## 3. 系统设计

整体采用分层设计：

- CLI 层：参数解析、命令分发、输出组织
- 状态层：session 和账号元数据
- 认证层：Playwright 驱动登录
- 资源层：Blackboard 页面抓取
- 媒体层：录播分片下载、解密、转封装、探测

这种结构的好处是：

- 页面抓取逻辑和 CLI 输出解耦
- 登录和资源读取可复用
- 后续更容易扩展 MCP 或 agent 集成

## 4. 为什么选 Playwright

教学网相关流程依赖：

- 北大统一认证
- Blackboard 的动态页面与跳转
- 鉴权后的播放器与资源

所以只用 `requests` 很难长期稳定。Playwright 更适合这里的真实场景。

## 5. 账号管理的演进

项目早期依赖工作区里的凭据文件，这种方式开发快，但不够规范，也不适合公开项目。  
现在改成：

- 终端交互输入
- Keychain 保存密码
- 本地 JSON 保存非敏感元数据
- 默认账号机制

这是用户体验和安全性上的明显提升。

## 6. 面向 agent 的设计

项目在 CLI 设计时明确考虑了 agent 使用：

- 命令结构稳定
- `--json` 输出稳定
- 支持活动课程上下文
- 默认不做高风险写操作
- 下载结果尽量结构化返回

这样可以降低 agent 猜课程名、误解析输出、误写入站点的概率。

## 7. 风险与限制

### Blackboard 页面结构变化

如果站点 DOM 有明显改动，抓取逻辑需要更新。

### 平台依赖

当前账号安全存储主要依赖 macOS Keychain，Linux / Windows 仍需要进一步兼容。

### 写操作测试受限

作业提交通常不能随便做 live 验证，因此写路径测试必须非常保守。

### 站点可用性

真实验证直接依赖教学网和北大统一认证的可用性。

## 8. 后续建议

- 增加非 macOS 的安全存储后端
- 为关键抓取器补充 HTML fixture 测试
- 增加更细的运行日志
- 为 agent 集成补 MCP 层
- 做正式版本化和 changelog
